03

Video Data Source

PurposeExplains where video playlists come from, how they are updated, and how they map to each page
PRD ReferenceChapter 2 — Data & Backend Logic
Why This Design

MVP scope descopes the video CMS backend — editors cannot manually update playlists, so all content maintenance must rely entirely on automated system scheduling.

News department's actual current update frequency:

  • Normal days: 2–5 times (9am, 12–1pm, 5–6pm, before midnight)
  • Major events: up to 10 times per day
Architecture Recommendation
Drop the tag-based dynamic search (Search API) in favour of fetching by YouTube Playlist ID directly — this is expected to avoid the risks of excessive API costs and Quota overflow.

Starting from the principle of "simple, fast, no rework," the recommendation is to adopt a system-scheduled periodic update approach, replacing manual intervention with a stable 1-hour cron schedule, rather than building a manual update interface for the news department.
Playlist Usage Scope (Five Pages)
Homepage
VOD 1+3 source — pulls the latest video from each Playlist
Video Hub
Editorial block source — each Playlist maps to one 1-large + 3-small block
Playlist Page
Tab label source, and the 24-video content source for each tab
Player Page
Initialises the 24-item navigation list using the video's Primary Playlist
Weather Page
Weather video block source — one dedicated Playlist, mounted by adding a backend tag only, no changes to playback logic required
Auto-Update Every Hour, 24 Articles per Playlist
  • System auto-schedules an update every 10 minutes
  • Each Playlist fetches the latest 24 articles
  • Within one refresh cycle, the playlist is fixed data
  • No manual force-update backend interface will be built
Playlist Name Auto-Filtering

YouTube Playlist names include a platform suffix. The system must automatically strip these for in-site UI display:

  • Remove the suffix │ TVBS News (including leading space and separator)
  • Remove leading and trailing 【 and 】 brackets from the title

In-site display: TVBS Daily Hot Search
YouTube original: 【Breaking News First Look】│ TVBS News
In-site display: Breaking News First Look

Key Reminder: The navigation list source is determined entirely by the playlist_id in the URL, not the entry point. video_id controls playback; playlist_id controls the list. They are independent.
playlist_id Mechanism: During the scheduled job, the backend maps each video to its corresponding playlist based on the YouTube API response and writes the playlist_id into the link. This value is then fixed. The Primary Playlist mapping concept has been removed — the list source is now entirely URL-driven.
Player Page Navigation List Display Rules:
  • Mapping exists: Show the current 24 items from the Primary Playlist, regardless of whether the current video is in the list
  • Mapping does not exist (video has rolled out of all Playlist current 24-item lists): Show the latest 24 videos across all Playlists sorted by published date descending as fallback — do not leave blank

※ Playlist display order is determined by the news department. Hardcoded in MVP stage — changes require a development request.

  • Desktop: Player is not sticky — scrolls away with the page as the user scrolls down

Player State Machine

Applies ToVideo Hub Hero Live, Single Video Player Page
STATE 1
Idle

Page entered, not yet playing

Show Overlay
Poster + Play Icon
STATE 2
Autoplay Attempt

autoplay(muted) triggered

Still showing Overlay
Prevents black screen flash
STATE 3
Playing

YT state = PLAYING

Hide Overlay
YouTube controls take over
STATE 4
Blocked / Failed

autoplay blocked or timed out

Show Overlay
Awaiting user tap

Overlay Composition (Minimum)

  • Video poster image as background
  • Centre-aligned Play Icon (tappable)
  • Coverage area: player viewport
  • Shown only when playback has not yet started
Overlay is an in-site DOM element
Sits outside the YouTube Iframe — does not modify the iframe internals.
On tap, calls YouTube API playVideo() to start playback.

Autoplay Core Rules

Autoplay is always muted Spec Defined
Unmuted playback requires manual user trigger Spec Defined
External direct link to Player Page — do not attempt autoplay Exception
Must avoid a black screen with no feedback Recommended
On playback success, remove Overlay Expected Behaviour

Error Handling & Fallback

Homepage Block API Failure

Affected: News Topics, News Boards, Category Focus, Video Section

→ Hide block directly (display:none)

No error message shown — layout collapses automatically

Taboola / Insider Load Failure

Script failure / API no response / AdBlock

→ Hide container directly, no blank space
Thumbnail Failure (YouTube)
Live Stream Block: Uses the Live-specific default image — does not follow the fallback logic below
Other blocks (VOD cards) fallback order:
Level 1: Video thumbnail (maxresdefault / hqdefault)
Level 2: Default image with Play Button

Triggered by listening to img onError event

Video Unavailable (Private / Deleted)

Listen to YouTube IFrame API onError (Code: 100, 101, 150)

→ Show YouTube default error screen. No additional UI guidance.

Navigation list below follows the same playlist_id validity rules as normal.

Hub Playlist API Failure
→ Hide the entire Playlist block (entire 1-large-3-small group)
Player Page Playlist API Failure
→ Show only the player and video info

Hide the 24-item navigation list below

Key Reminders

PurposePre-session preparation / Quick reference during session / Post-session confirmation

⚠ Common Points of Confusion

FE The Play Button in the Homepage video block drives traffic — it does not play in-page
Tapping navigates to /video/{video_id}. The Homepage does not support in-page playback and does not autoplay.
FE News Boards Slot 1–4 all behave the same — the LIVE tag is a visual indicator only
All Slots are CMS image + link. When a live URL is set in CMS, the frontend displays a red LIVE tag at the top-left of that Slot. Tap behaviour is unchanged — navigates to the CMS-configured link.
FE The first item in the Player Page list does not show a Play Button — only a highlighted background
Items #2–24 show the Play Button. Item #1 is still tappable — tapping reloads to re-enter the same video.
FE BE The Playlist Page's /playlist/{id} is not carried into the Player Page
The Player Page ignores the source parameter entirely — it always initialises the navigation list using the video's Primary Playlist.
FE The 24 items on the Player Page form a navigation list — not a playback queue
After the video ends, playback stops at the end frame. It does not auto-advance. Users must tap the list to navigate to a new video.
FE The Overlay is an in-site DOM element — it is not inside the YouTube iframe
On tap, it calls the IFrame Player API's playVideo(). The Overlay is only hidden after playback successfully begins.
BE The Realtime News backend returns only 8 articles — ad slots are injected by the frontend
The frontend inserts ads at positions 3 and 8. This does not affect the backend's article count or dedup logic.

🔴 Things Easily Overlooked During Implementation

BE Category Focus backfill: Carousel articles must never be used as backfill under any circumstances
Backfill priority: ① Unused articles from the same category → ② Articles already in Realtime News. Backfill each category to 4 articles.
FE The Overlay must remain visible throughout the Autoplay Attempt phase — do not remove it early
Hide only when the YouTube Player state enters PLAYING. The Overlay must remain visible during the Attempt phase.
FE Playlist Page tab switching is a full page navigation — MVP does not use AJAX
Tapping a tab navigates to /playlist/{id} with a full page reload. No frontend routing required.
