02

首頁內容排重邏輯 — MVP 階段，開發評估暫時不做

目的避免首頁連續區塊出現相同新聞
原則最低工程成本，僅針對三個最易重複的區塊
1

News Carousel（輪播）— 最高優先，不受排重影響

編輯手動策展，內容不因任何排重規則調整。
→ 建立 carousel_news_ids 集合，供後續區塊排重使用。

2

Realtime News（即時新聞）

抓取最新新聞時，先排除 carousel_news_ids。
實際顯示 8 則，廣告洞位在第3、8則位置插入（廣告不影響新聞索引）。
→ 建立 realtime_news_ids（僅8則新聞本身，不含廣告位）

// 顯示順序 新聞 #1 新聞 #2 [AD 廣告洞位] ← 版面插入，不計入 realtime_news_ids 新聞 #3 新聞 #4 新聞 #5 新聞 #6 [AD 廣告洞位] ← 版面插入，不計入 realtime_news_ids 新聞 #7 新聞 #8
3

Category Focus（分類焦點）

抓取時排除 carousel_news_ids + realtime_news_ids。
每分類目標顯示 4 則，不允許少於4則（不顯示空白）。

補位規則（優先順序）
1. 補同分類中未用過的新聞
2. 若仍不足，允許補入已在 Realtime News 的新聞
3. 任何情況下，不補入已在 Carousel 的新聞
不參與排重的區塊：News Topics, News Boards, Featured Topics, Video Section, Taboola模組。
03

影音資料來源

用途說明影音清單從哪裡來、如何更新、怎麼對應到各頁面
對應 PRD第 2 章 — 資料與後端邏輯
為什麼這樣設計

MVP 階段 Descope 不建立影音後台，編輯無法手動更新清單，內容維護必須全面改為系統自動化排程。

網新現有的實際更新頻率：

  • 平時一天 2–5 次（早上 9 點、中午 12–1 點、下午 5–6 點、晚上 12 點前）
  • 大事件時一天最多 10 次
架構考量建議
捨棄依賴 Tag 的動態搜尋（Search API），改採 YouTube Playlist ID 指定抓取，預期上可以避免 API 成本過高與 Quota 爆量的風險。

從「簡單、快速、不重工」出發思考，建議採用系統每 10 分鐘定期更新的作法，以系統穩定排程取代人工介入，不去思考額外需要提供網新手動更新的途徑。

  • 系統每 10 分鐘自動排程更新一次

關鍵提醒：播放頁的導覽清單來源完全由 URL 的 playlist_id 決定，與入口無關。video_id 管播放，playlist_id 管清單，兩者獨立互不影響。
playlist_id 機制：後端排程執行時，依 YouTube API 抓取結果將影片對應到所屬 playlist，並將 playlist_id 寫入連結，之後固定不變。Primary Playlist mapping 概念已移除，清單來源改為純 URL 驅動。
播放頁導覽清單顯示規則：
  • 有效 playlist_id：顯示該 playlist 當下最新24則（與影片是否在其中無關）
  • 無 playlist_id：四個 playlist 合併依 published_at 降冪取最新24則
  • 無效 playlist_id：下方清單區塊空白，不 render

※ Playlist 顯示順序由新聞部指定，MVP 階段後端 hardcode，調整需走開發流程。

播放成功後移除 Overlay 預期行為

錯誤處理與降級機制

首頁區塊 API 失敗

受影響：News Topics, News Boards, Category Focus, Video Section

→ 直接隱藏（display:none）

不顯示錯誤訊息，版面自動塌陷

Taboola / Insider 載入失敗

Script 失敗 / API 無回應 / AdBlock 阻擋

→ 直接隱藏容器，不留白
縮圖失效（YouTube）
Live 直播區塊：使用 Live 專屬預設圖片，不走下方降級邏輯
其他區塊（VOD 卡片）降級順序：
Level 1: 影片縮圖（maxresdefault / hqdefault）
Level 2: Default image 帶 Play Button

監聽 img onError 事件觸發替換

影片失效（Private / Deleted）

監聽 YouTube IFrame API onError（Code: 100, 101, 150）

→ 顯示 YouTube 預設錯誤畫面，不加額外 UI 引導

下方清單邏輯不變，依 URL 的 playlist_id 有效性決定顯示內容。

Hub Playlist API 失敗
→ 隱藏該 Playlist 區塊（1大3小整組隱藏）

→ 僅顯示播放器與影片資訊

重點提醒

用途說明會前預習 / 說明會中即時查閱 / 說明會後確認

⚠ 容易混淆的地方

FE 首頁影音區塊的 Play Button 是導流，不是播放
點擊後跳轉至 /video/{video_id}?playlist={playlist_id}，首頁不支援頁內播放也不做 Autoplay
FE 新聞看板 Slot 1–4 行為一致，LIVE tag 只是視覺標示
所有 Slot 皆為 CMS 圖片＋連結。CMS 設定直播 URL 時，前端在該 Slot 左上角加上紅色 LIVE tag，點擊行為不變，導向 CMS 設定的連結
FE BE Playlist 列表頁的 playlist_id 會帶進播放頁
從列表頁點入時前端帶入 ?playlist={id}，播放頁下方清單依此顯示。直接輸入或舊連結無此參數時，顯示四列表合併最新24則
FE Overlay 是站內 DOM，不在 YouTube iframe 內
點擊後透過 IFrame Player API 呼叫 playVideo()，播放成功才隱藏 Overlay
BE 即時新聞後端只回 8 則，廣告洞位由前端插入
前端在第 3、第 8 個位置插入廣告，不影響後端數量與排重邏輯

🔴 容易在實作中被忽略

BE Category Focus 補位：任何情況下不得補入已在 Carousel 的新聞
補位順序：① 同分類未使用 → ② 已在 Realtime News 的可補。各分類補齊至 4 則
FE Autoplay 失敗期間 Overlay 必須持續顯示，不能提前移除
直到 YouTube Player state 進入 PLAYING 才隱藏，Attempt 階段也要保持顯示
FE Playlist 列表頁 Tab 切換是整頁跳轉，MVP 不使用 AJAX
點擊 Tab 後導向 /playlist/{id}，整頁重新載入，不需前端路由
